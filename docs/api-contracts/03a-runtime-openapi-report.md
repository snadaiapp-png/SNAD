## 1. Objective

Generate the OpenAPI contract for the SNAD platform FROM THE RUNNING APPLICATION (not a hand-written YAML) and use it as the authoritative baseline for API compatibility enforcement.

## 2. Approach — Method B (SpringBootTest + MockMvc)

The OpenAPI document is generated by:

1. Booting the full Spring application context in the `local` profile (H2 in-memory DB, no external dependencies).
2. Springdoc-openapi (v2.6.0) auto-discovers all `@RestController` classes and their `@Operation` / `@ApiResponse` annotations.
3. The test issues `GET /v3/api-docs` via MockMvc.
4. The response body is validated for structural integrity (minimum paths, schemas, all paths start with `/api/v1/` or `/actuator`).
5. The validated JSON is written to `apps/sanad-platform/build/api-contract/openapi-current.json`.

## 4. CI Integration

The `api-contract` job in `.github/workflows/quality-gate.yml` runs:

1. `OpenApiContractExportTest` — generates `openapi-current.json` from the running app.
2. Validates the generated file is non-trivial (>5 KB).
3. Runs `oasdiff breaking` against the committed baseline.
4. Uploads both the current OpenAPI and the diff report as CI artifacts.
5. Verifies the committed API inventory (`03-api-inventory.json`) path list matches the runtime path list.

## 5. Manual Baseline Update

To regenerate the baseline after an INTENTIONAL non-breaking change:

```bash
bash scripts/dev/capture-openapi-baseline.sh
git add docs/api-contracts/openapi-v1-baseline.yaml docs/api-contracts/openapi-v1-baseline-metadata.json
git commit -m "chore(api): update OpenAPI v1 baseline after non-breaking change"
```

The CI will FAIL on any breaking change vs. the committed baseline; it will never auto-update the baseline.
