# قرار قبول المخاطر المؤقت — SANAD

<!-- STATUS_AUTHORITY: SUPPORTING_DECISION -->

**تاريخ النفاذ:** 18 يوليو 2026 — Asia/Riyadh  
**صاحب القرار:** Project Owner  
**سجل المتابعة الحاكم:** GitHub Issue #516  
**طبيعة القرار:** قبول مخاطر مؤقت ومشروط، وليس إغلاقًا للملاحظات

## 1. القرار

يقبل Project Owner مؤقتًا المخاطر المتبقية التالية ضمن نطاق محدود فقط:

- `REM-P0-001` — اعتماد الباكند الإنتاجي على نفق تطوير مؤقت.
- `REM-P0-002` — عدم اكتمال قبول موثوقية BFF والمصادقة والجلسات.
- `REM-P0-004` — عدم تسوية تسلسل الحوكمة للمخرجات اللاحقة.
- `REM-P0-005` — عدم اكتمال إثبات النسخ الاحتياطي والاستعادة والتعافي من الكوارث.
- `REM-P0-006` — عدم اكتمال الضمان الأمني المستقل.
- `REM-P1-009` — عدم حسم خصوصية مستودع GitHub.

```text
TEMPORARY_RISK_ACCEPTANCE: ACTIVE_FOR_CONTROLLED_SCOPE_ONLY
PROJECT_STATUS: CONDITIONAL_CONTINUE
CONTROLLED_DEVELOPMENT: ALLOWED
VERIFICATION: ALLOWED
LIMITED_PILOT: ALLOWED
BROAD_COMMERCIAL_GO_LIVE: NOT_APPROVED
OPEN_FINDINGS: 6
```

## 2. النطاق المسموح

يسري القبول المؤقت فقط على:

1. التطوير المنضبط داخل المشروع.
2. الاختبارات والتحقق وإعادة الاختبار.
3. التشغيل التجريبي المحدود وغير التعاقدي.
4. أعمال الإصلاح والترحيل والتحسين التشغيلي.
5. جمع الأدلة اللازمة لإغلاق الملاحظات المتبقية.

## 3. النطاق غير المشمول

لا يسمح هذا القرار بما يلي:

1. الإطلاق التجاري الواسع أو تقديم المنصة بوصفها جاهزة مؤسسيًا بالكامل.
2. تقديم التزامات SLA خارجية أو تعاقدية.
3. قبول أحمال عالية الحساسية أو بيانات منظمة خارج حدود التجربة المعتمدة.
4. إسقاط شدة أي ملاحظة أو تغيير حالتها إلى `CLOSED`.
5. تجاوز بوابات الأمن أو الاستمرارية أو الجاهزية الإنتاجية.
6. اعتبار نجاح CI أو فحص منفرد دليلًا على قبول الإنتاج الموسع.

## 4. الضوابط التعويضية الإلزامية

طوال مدة القبول المؤقت يجب استمرار الضوابط التالية:

- تشغيل المراقبة والتنبيهات وسجلات الحوادث دون تعطيل.
- إبقاء بوابات CI والأمن والجاهزية Fail-Closed.
- عدم توسيع نطاق التجربة أو عدد المستخدمين أو حساسية البيانات دون مراجعة جديدة.
- استمرار النسخ الاحتياطي المتاح حاليًا مع توثيق أي فشل أو فجوة.
- عدم إخفاء أخطاء `502/503/504` أو استبعادها من تقارير الخدمة.
- فتح Incident عند تحقق أي أثر جوهري للمخاطر المقبولة.
- الاحتفاظ بقابلية Rollback وخطة استعادة لكل تغيير إنتاجي.
- مراجعة المخاطر المقبولة كل 30 يومًا وتسجيل نتيجة المراجعة في Issue #516.

## 5. الإيقاف والانتهاء التلقائي

ينتهي أو يتوقف هذا القبول تلقائيًا عند أول حالة من الحالات التالية:

1. بدء مراجعة أو قرار الإطلاق التجاري الواسع.
2. عدم تسجيل مراجعة دورية خلال 30 يومًا.
3. وقوع حادث `SEV0` أو `SEV1` مرتبط بإحدى المخاطر المقبولة.
4. حدوث اختراق مؤكد، تسرب بيانات، فقد بيانات، أو تجاوز لعزل المستأجرين.
5. تغير جوهري في الاستضافة أو الهوية أو الأمن أو نموذج البيانات.
6. توسع التشغيل التجريبي خارج الحدود المسجلة في هذا القرار.
7. طلب Security Governance أو Infrastructure أو Data Platform إيقاف القبول بسبب ارتفاع التعرض.

عند انتهاء القبول، تعود البنود فورًا إلى كونها موانع تشغيلية غير مقبولة إلى أن يصدر قرار جديد أو تكتمل شروط الإغلاق.

## 6. أثر القرار على كل ملاحظة

| الملاحظة | الحالة بعد القرار | أثر القبول المؤقت |
|---|---|---|
| REM-P0-001 | `OPEN / DEFERRED / TEMPORARILY_ACCEPTED` | يسمح بالتطوير والتجربة المحدودة فقط؛ لا يعتمد النفق كاستضافة نهائية |
| REM-P0-002 | `OPEN / REMEDIATED_BUT_UNACCEPTED / TEMPORARILY_ACCEPTED` | لا يلغي نافذة المراقبة أو اعتماد REM-P0-001 |
| REM-P0-004 | `OPEN / TEMPORARILY_ACCEPTED` | لا يحل تعارضات الحوكمة ولا يعتمد المخرجات تلقائيًا |
| REM-P0-005 | `OPEN / TEMPORARILY_ACCEPTED` | لا يثبت RPO/RTO أو DR ولا يسمح بادعاء الاستمرارية |
| REM-P0-006 | `OPEN / TEMPORARILY_ACCEPTED` | لا يستبدل التقييم الأمني المستقل أو قبول Security Governance |
| REM-P1-009 | `OPEN / TEMPORARILY_ACCEPTED` | لا يحسم قرار `PUBLIC` أو `PRIVATE` ولا يلغي المراجعة القانونية والأمنية |

## 7. شروط الانتقال إلى الإنتاج الموسع

قبل أي موافقة على الإنتاج الموسع يجب:

1. إعادة فتح مراجعة المخاطر الستة بصورة مستقلة.
2. إغلاق كل ملاحظة بأدلة قابلة لإعادة الإنتاج، أو إصدار قبول متبقٍ منفصل وصريح من الجهات المخولة.
3. نجاح بوابة Production Readiness على الإصدار المرشح.
4. اعتماد Security Governance وInfrastructure وData Platform وQA & Release بحسب نطاق كل ملاحظة.
5. توثيق RPO/RTO وSLO والقدرة الاستيعابية وخطة Rollback.
6. تحديث `CURRENT-STATUS.json` وIssue #516 قبل القرار التجاري.

## 8. الحكم النهائي

هذا القرار يسمح باستمرار العمل المنضبط ولا يغيّر حقيقة أن المخاطر الستة ما زالت مفتوحة. المسؤولية عن آثارها مقبولة مؤقتًا من Project Owner ضمن الحدود المذكورة فقط.
