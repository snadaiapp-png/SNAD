## 3. الأخطاء الحرجة غير المصححة — P0

### REM-P0-001 — اعتماد الباكند الإنتاجي على نفق تطوير مؤقت

**الحالة:** `DEFERRED / OPEN / NOT ACCEPTED AS FINAL HOSTING`

#### الوضع الحالي

يعتمد ربط الإنتاج بالباكند على نفق تطوير مؤقت، وليس على استضافة إنتاجية مُدارة ذات نطاق ثابت وضمانات تشغيلية مؤسسية.

#### المخاطر

- انقطاع الخدمة عند توقف النفق أو الجهاز أو العملية المضيفة.
- تغير العنوان أو انتهاء النفق.
- غياب ضمانات التوافر والسعة والتوسع.
- ضعف التحكم في الشبكات والأسرار والـTLS والـRollback.
- صعوبة إثبات الامتثال والاستمرارية التشغيلية.

#### شروط الإغلاق

1. نشر الباكند في منصة إنتاج مُدارة.
2. استخدام نطاق HTTPS ثابت ومملوك للمشروع.
3. إدارة الأسرار خارج المستودع.
4. توفير مراقبة وسجلات وتنبيهات مركزية.
5. اعتماد حدود السعة والتوسع والتوافر.
6. توفير Rollback واختبار Cutover منظم.
7. فصل Development وStaging وProduction.
8. إثبات نجاح المسارات الإنتاجية بعد الانتقال.

### REM-P0-002 — عدم استقرار BFF والمصادقة والجلسات

**الحالة:** `DEFERRED / OPEN` حيث يعتمد الإصلاح على معالجة الباكند والنفق.

#### الوضع الحالي

ظهرت استجابات صحيحة مثل `HTTP 401` للمستخدم غير المسجل، لكن ظهرت أيضًا مهلات وأخطاء متقطعة مثل `502` و`503` في مسارات المصادقة والجلسات.

#### المخاطر

- فشل تسجيل الدخول أو استعادة الجلسة بصورة متقطعة.
- فشل Token Refresh أو Logout.
- خروج المستخدم أو فقد العمليات غير المحفوظة.
- عدم إمكانية ضمان الوصول المؤسسي المستمر.
- استهلاك Error Budget وفتح حوادث تشغيلية.

#### شروط الإغلاق

1. إزالة اعتماد المسار على نفق التطوير.
2. ضبط Connection وRead Timeouts والسعة.
3. تشغيل Synthetic Journey يغطي Login وSession Restore وRefresh وLogout وLockout وAudit.
4. تحديد نافذة مراقبة معتمدة دون أخطاء `502/503` غير مبررة.
5. إثبات معدلات النجاح والكمون على إصدار وبيئة محددين.
6. اعتماد النتيجة من Identity وOperations.

### REM-P0-004 — عدم تسوية تسلسل الحوكمة للمخرجات اللاحقة

**الحالة:** `OPEN`

#### المشكلة

تم إنتاج مخرجات معمارية وتشغيلية لاحقة خلال فترة كان فيها Gate المنفذ #23 مغلقًا. لم يصدر حتى الآن قرار موحد يصنف كل مخرج لاحق رسميًا.

#### المخاطر

- تعارض مصادر القرار والنسخ المعتمدة.
- اعتماد قرارات معمارية أو تشغيلية دون تفويض واضح.
- بناء أعمال لاحقة على مخرجات قد تكون مسودة أو استثناءً غير موثق.
- ضعف قابلية التدقيق والمساءلة.

#### شروط الإغلاق

1. إصدار قرار مؤرخ من Steering Committee.
2. تصنيف كل مخرج إلى `DRAFT` أو `EXCEPTION-AUTHORIZED` أو `APPROVED` أو `REJECTED`.
3. ربط القرار بالوثيقة والنسخة والـSHA.
4. تحديد مالك القرار وتاريخ النفاذ.
5. تحديث جميع سجلات الحالة وحذف التعارضات.

### REM-P0-005 — عدم اكتمال إثبات النسخ الاحتياطي والاستعادة والتعافي من الكوارث

**الحالة:** `OPEN`

#### المشكلة

لا توجد حزمة أدلة إنتاجية مقبولة تثبت النسخ الاحتياطي المشفر، والاستعادة المعزولة، وتحقيق RPO/RTO، وتنفيذ تمرين Disaster Recovery كامل على البيئة الحالية.

#### المخاطر

- فقد دائم أو طويل الأثر للبيانات.
- فقد القيود المحاسبية وبيانات العملاء والموظفين.
- توقف ممتد دون خطة تعافٍ مثبتة.
- عدم الوفاء بالالتزامات التعاقدية والتنظيمية.

#### شروط الإغلاق

1. اعتماد RPO وRTO لكل خدمة حرجة.
2. تفعيل نسخ آلية ومشفرة وسياسة Retention.
3. فصل نسخ الاستعادة عن البيئة الأساسية.
4. تنفيذ Restore في بيئة معزولة.
5. التحقق من تكامل البيانات وتشغيل التطبيق على النسخة المستعادة.
6. تنفيذ DR Drill وقياس الزمن الفعلي.
7. توثيق النتائج والفجوات واعتمادها رسميًا.

### REM-P0-006 — عدم اكتمال الضمان الأمني المستقل

**الحالة:** `OPEN`

#### المشكلة

توجد اختبارات وفحوص داخلية، لكنها لا تشكل ضمانًا أمنيًا مستقلًا للبيئة الإنتاجية الكاملة.

#### الأعمال غير المكتملة

- اختبار اختراق مستقل.
- اختبار Tenant Isolation وObject-Level Authorization.
- مراجعة إعدادات الإنتاج والأسرار والصلاحيات.
- مراجعة الخصوصية ونموذج التهديدات والتبعيات.
- إعادة اختبار الإصلاحات وقبول المخاطر المتبقية.

#### المخاطر

- تجاوز الصلاحيات أو حدود المستأجرين.
- كشف بيانات أو أسرار حساسة.
- تعديل بيانات مالية أو تشغيلية دون تفويض.
- آثار قانونية ومالية وسمعة سلبية.

#### شروط الإغلاق

1. تنفيذ التقييمات المستقلة المطلوبة.
2. معالجة جميع الثغرات الحرجة والعالية.
3. إعادة اختبار الإصلاحات على إصدار محدد.
4. توثيق المخاطر المتبقية ومالكيها.
5. اعتماد Security Governance وProject Owner.

## 4. الأخطاء المرتفعة غير المصححة — P1

### REM-P1-007 — عدم اكتمال إثبات العمليات التجارية المتكاملة E2E

**الحالة:** `OPEN`

#### المشكلة

لم تُقبل حزمة موحدة تثبت العمليات العابرة للموديولات من البداية إلى النهاية مع التأكيدات المالية والمخزنية والأمنية والتدقيقية.

#### السيناريوهات المطلوبة

- Lead → Opportunity → Order → Delivery → Invoice → Ledger → Collection.
- Purchase Request → Approval → PO → Receipt → Supplier Invoice → Payment.
- Employee → Attendance/Leave → Payroll → Accounting Posting → Payment.
- Ecommerce Order → Payment → Inventory → Shipment → Return/Refund → Reconciliation.
- Workflow approvals، tenant isolation، audit trail، analytics، rollback والتعافي من الفشل.

#### المخاطر

- نجاح الوحدات منفردة وفشل العملية التجارية الكاملة.
- قيود محاسبية أو حركات مخزون غير صحيحة.
- فقد أو ازدواج المعاملات.
- تقارير وتحليلات غير متسقة.

#### شروط الإغلاق

1. تعريف السيناريوهات الحرجة وربطها بالمتطلبات.
2. إعداد بيانات اختبار معروفة ومتعددة المستأجرين.
3. إثبات الصلاحيات والعزل والتدقيق.
4. إثبات النتائج المالية والمخزنية والـRollback.
5. تشغيل Regression Suite على SHA محدد.
6. اعتماد ملاك العمليات وQA & Release.

### REM-P1-009 — عدم حسم خصوصية مستودع GitHub

**الحالة:** `OPEN`

#### الوضع الحالي

المستودع ظاهر حاليًا بوصفه `public`، بينما تضمنت وثائق سابقة وصفًا مختلفًا. لم يُسجل قرار حوكمي نهائي يثبت أن هذا التعرض مقصود ومقبول بعد مراجعة أمنية وقانونية.

#### المخاطر

- كشف البنية والمسارات ونماذج البيانات والوثائق التجارية.
- احتمال وجود أسرار أو بيانات حساسة في Current Tree أو Git History.
- تعارض مع سياسات الملكية الفكرية والسرية.
- توسيع سطح التحليل والهجوم.

#### شروط الإغلاق

1. إصدار قرار رسمي: `PUBLIC` أو `PRIVATE`.
2. فحص Current Tree وGit History للأسرار والمعلومات الحساسة.
3. تدوير أي سر مكشوف وإزالة المحتوى غير المسموح.
4. مراجعة قانونية وملكية فكرية.
5. تحديث كل الوثائق والإعدادات المتعارضة.
6. تسجيل صاحب القرار وتاريخه وأدلته.
