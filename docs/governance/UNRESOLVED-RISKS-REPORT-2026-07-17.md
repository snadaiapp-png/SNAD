## 1. الغرض والنطاق

هذا التقرير هو المرجع الحالي للبنود التي لم تُغلق بعد. لا يتضمن البنود المغلقة، ولا يجوز استخدام تقارير المراحل التاريخية أو نتيجة فحص منفردة لتجاوز الحالة الواردة هنا.

توجد ستة بنود مفتوحة:

- خمسة بنود P0 حرجة.
- بند P1 واحد مرتفع.
- `REM-P0-001` مؤجل بقرار مالك المشروع وما زال مفتوحًا.
- نُفذت ضوابط التطبيق الخاصة بـ`REM-P0-002`، لكنه ما زال مفتوحًا حتى اكتمال النشر والمراقبة ومعالجة اعتماده على `REM-P0-001`.
- قُبلت المخاطر الستة مؤقتًا ضمن التطوير المنضبط والتحقق والتشغيل التجريبي المحدود فقط.
- القبول المؤقت لا يغلق أي بند ولا يخفض شدته ولا يسمح بالإطلاق التجاري الواسع.

مرجع قرار القبول المؤقت:

`docs/governance/TEMPORARY-RISK-ACCEPTANCE-2026-07-18.md`

## 3. حدود قبول المخاطر المؤقت

يسمح القبول المؤقت فقط بـ:

1. التطوير المنضبط.
2. الاختبارات والتحقق وإعادة الاختبار.
3. التشغيل التجريبي المحدود وغير التعاقدي.
4. تنفيذ أعمال الإصلاح والترحيل وجمع الأدلة.

ولا يسمح بـ:

1. الإطلاق التجاري الواسع.
2. تقديم التزامات SLA تعاقدية.
3. إعلان الجاهزية المؤسسية الكاملة.
4. توسيع نطاق المستخدمين أو حساسية البيانات دون مراجعة جديدة.
5. إغلاق البنود أو خفض شدتها دون استيفاء شروط الإغلاق.

تجب مراجعة القرار كل 30 يومًا. وينتهي تلقائيًا عند بدء مراجعة الإنتاج الموسع، أو عند حادث `SEV0/SEV1` مرتبط بالمخاطر، أو اختراق مؤكد، أو فقد بيانات، أو تجاوز لعزل المستأجرين، أو تغير معماري جوهري، أو توسع التجربة خارج الحدود المعتمدة.

## 4. الأخطاء الحرجة غير المصححة — P0

### REM-P0-001 — اعتماد الباكند الإنتاجي على نفق تطوير مؤقت

**الحالة:** `DEFERRED / OPEN / TEMPORARILY_ACCEPTED / NOT ACCEPTED AS FINAL HOSTING`

يعتمد ربط الإنتاج بالباكند على نفق تطوير مؤقت، وليس على استضافة إنتاجية مُدارة ذات نطاق ثابت وضمانات تشغيلية مؤسسية. القبول المؤقت يسمح باستمرار العمل المحدود فقط، ولا يعتمد النفق كاستضافة نهائية.

**شروط الإغلاق:**

1. نشر الباكند في منصة إنتاج مُدارة.
2. استخدام نطاق HTTPS ثابت ومملوك للمشروع.
3. إدارة الأسرار خارج المستودع.
4. توفير مراقبة وسجلات وتنبيهات مركزية.
5. اعتماد حدود السعة والتوسع والتوافر.
6. توفير Rollback واختبار Cutover.
7. فصل Development وStaging وProduction.
8. إثبات نجاح المسارات الإنتاجية بعد الانتقال.

---

### REM-P0-002 — عدم استقرار BFF والمصادقة والجلسات

**الحالة:** `APPLICATION CONTROLS IMPLEMENTED / OPEN / TEMPORARILY_ACCEPTED / PENDING PRODUCTION OBSERVATION AND REM-P0-001`

نُفذت ميزانية مهلة BFF الكلية، وإعادة المحاولة المقيدة للطلبات الآمنة، وRefresh Single-Flight، وسلامة Cookies، ورحلة Synthetic للإنتاج. مرجع الضوابط هو `docs/operations/reliability/AUTH-SESSION-RELIABILITY.md`.

**المتبقي للإغلاق:**

1. نشر SHA المحدد في الإنتاج.
2. نجاح الرحلة الكاملة لهوية Synthetic المحمية.
3. اجتياز 72 دورة متتالية كل ساعة.
4. غياب أخطاء `502/503/504` غير المفسرة.
5. إثبات Login وSession Restore وRefresh وLogout وLockout وAudit.
6. إصدار تقرير التوافر والكمون واستهلاك Error Budget.
7. إكمال PIR لأي حادث داخل النافذة.
8. اعتماد Identity وPlatform Operations وInfrastructure.
9. إزالة اعتماد `REM-P0-001` أو قبوله رسميًا كمخاطر متبقية عند بوابة الإنتاج الموسع.

---

### REM-P0-004 — عدم تسوية تسلسل الحوكمة للمخرجات اللاحقة

**الحالة:** `OPEN / TEMPORARILY_ACCEPTED`

تم إنتاج مخرجات لاحقة خلال فترة كان فيها Gate المنفذ #23 مغلقًا، ولم يصدر قرار موحد يصنف كل مخرج. القبول المؤقت لا يعتمد هذه المخرجات تلقائيًا.

**شروط الإغلاق:**

1. قرار مؤرخ من Steering Committee.
2. تصنيف كل مخرج إلى `DRAFT` أو `EXCEPTION-AUTHORIZED` أو `APPROVED` أو `REJECTED`.
3. ربط القرار بالوثيقة والنسخة والـSHA.
4. تحديد مالك القرار وتاريخ النفاذ.
5. تحديث جميع سجلات الحالة وإزالة التعارضات.

---

### REM-P0-005 — عدم اكتمال إثبات النسخ الاحتياطي والاستعادة والتعافي

**الحالة:** `OPEN / TEMPORARILY_ACCEPTED`

لا توجد حزمة أدلة إنتاجية مقبولة تثبت النسخ الاحتياطي المشفر، والاستعادة المعزولة، وتحقيق RPO/RTO، وتمرين Disaster Recovery كامل. القبول المؤقت لا يثبت استمرارية الأعمال ولا يسمح بادعائها.

**شروط الإغلاق:**

1. اعتماد RPO وRTO لكل خدمة حرجة.
2. تفعيل نسخ آلية ومشفرة وسياسة Retention.
3. فصل نسخ الاستعادة عن البيئة الأساسية.
4. تنفيذ استعادة فعلية مع فحوص الاتساق.
5. تنفيذ تمرين DR موثق.
6. اعتماد Infrastructure وData Platform وSecurity Governance.

---

### REM-P0-006 — عدم اكتمال الضمان الأمني المستقل

**الحالة:** `OPEN / TEMPORARILY_ACCEPTED`

بوابات CI الأمنية لا تستبدل SAST وDAST وفحص صور الحاويات واختبار الاختراق ومراجعة Tenant Isolation من جهة مستقلة. قبول Project Owner للمخاطر لا يستبدل اعتماد Security Governance.

**شروط الإغلاق:**

1. تنفيذ التقييمات المستقلة المطلوبة.
2. معالجة جميع الثغرات الحرجة والعالية.
3. إعادة اختبار الإصلاحات على إصدار محدد.
4. توثيق المخاطر المتبقية ومالكيها.
5. اعتماد Security Governance وProject Owner.

## 5. الأخطاء المرتفعة غير المصححة — P1

### REM-P1-009 — عدم حسم خصوصية مستودع GitHub

**الحالة:** `OPEN / TEMPORARILY_ACCEPTED`

المستودع ظاهر بوصفه `public`، بينما تضمنت وثائق سابقة وصفًا مختلفًا. لم يُسجل قرار نهائي يثبت أن هذا التعرض مقصود ومقبول بعد مراجعة أمنية وقانونية. القبول المؤقت لا يحسم قرار `PUBLIC` أو `PRIVATE`.

**شروط الإغلاق:**

1. إصدار قرار رسمي: `PUBLIC` أو `PRIVATE`.
2. فحص Current Tree وGit History للأسرار والمعلومات الحساسة.
3. تدوير أي سر مكشوف وإزالة المحتوى غير المسموح.
4. مراجعة قانونية وملكية فكرية.
5. تحديث الوثائق والإعدادات المتعارضة.
6. تسجيل صاحب القرار وتاريخه وأدلته.

## 6. البنود المغلقة والمستبعدة

البنود التالية مغلقة ولا تُعد أخطاء غير مصححة:

- `REM-P0-003` — Executor #23 Master Execution Backlog.
- `REM-P1-007` — إثبات العمليات التجارية المتكاملة E2E؛ أربع عمليات `FULLY_VERIFIED` مع PostgreSQL والتسويات والعزل والصلاحيات والتدقيق والـRollback.
- `REM-P1-008` — SLA/SLO ونموذج إدارة الحوادث.
- `REM-P1-010` — حوكمة وثائق الحالة وتصنيف السجلات التاريخية.
