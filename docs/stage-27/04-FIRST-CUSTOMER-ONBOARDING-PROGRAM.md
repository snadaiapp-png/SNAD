# Stage 27 — First Customer Onboarding Program

## Purpose

This document defines the onboarding program for SNAD's first customer under Stage 27.

The program ensures that the first customer is onboarded through a controlled, measurable, and supportable process covering discovery, tenant setup, user enablement, workflow configuration, UAT, go-live readiness, and post-go-live support.

## Onboarding Principles

```text
Start with controlled scope.
Define success criteria before configuration.
Use least privilege.
Protect customer data.
Do not activate production billing without separate approval.
Do not bypass partner or security controls.
```

## Onboarding Phases

| Phase | Outcome |
|---|---|
| Pre-onboarding | Customer scope and readiness confirmed |
| Kickoff | Roles, timeline, and success criteria aligned |
| Requirements | Processes, users, modules, and workflows documented |
| Tenant setup | Organization and baseline configuration prepared |
| User setup | Users, roles, RBAC, and access rules configured |
| Workflow setup | First workflows documented and prepared |
| Training | Customer users trained on initial scope |
| UAT | Customer validates pilot scenarios |
| Go-live readiness | Launch decision checklist completed |
| Post-go-live support | Support model activated |

## Kickoff Agenda

1. Confirm executive sponsor
2. Confirm implementation owner
3. Confirm partner role
4. Confirm pilot scope
5. Confirm success criteria
6. Confirm data/security constraints
7. Confirm timeline
8. Confirm support path

## Requirements Collection

Required discovery fields:

```text
Organization profile
Departments
Users
Roles
Approval workflows
Target modules
Current systems
Data migration needs
Reporting needs
Compliance constraints
Training needs
```

## User and Role Setup

Initial roles should include:

- Owner/Admin
- Manager
- Employee/User
- Finance user if needed
- HR user if needed
- CRM/Sales user if needed
- Partner implementation user only if approved

## Workflow Setup

Initial workflow candidates:

- Customer approval process
- Internal request process
- Sales follow-up process
- Finance approval process
- HR request process

All workflows must be limited to confirmed pilot scope.

## UAT Plan

UAT must validate:

- Login and access
- Role permissions
- Tenant isolation
- Core navigation
- Workflow execution
- Notification behavior
- Reporting output
- Support escalation path

## Go-Live Readiness Checklist

```text
Pilot scope confirmed
Users configured
Roles configured
Workflows configured
Training completed
UAT accepted
Support path ready
Rollback/support plan ready
No critical security blockers
No unresolved data authorization issues
```

## Post-Go-Live Support

Support must track:

- Issues by severity
- Time to first response
- Time to resolution
- Adoption blockers
- Training gaps
- Product feedback
- Expansion signals

## Success Metrics

```text
Time to onboard
UAT completion
Workflow adoption
User activation
Support ticket trend
Customer health score
Pilot success progress
```
